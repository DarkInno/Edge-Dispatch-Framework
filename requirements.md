# 需求文档（PRD）：Edge Dispatch Framework

> 版本：v0.1（草案）  
> 默认接入方式：**302 调度**（HTTP 重定向）  
> 核心理念：**Anycast（可选入口增强）+ Edge（边缘节点）**，用统一调度协议把用户请求分发到更合适的边缘节点，并提供可观测与治理能力。

---

## 1. 背景与问题

在多地域访问、跨运营商、弱网等场景下，用户直连源站/单地域服务通常会遇到：
- 延迟高、抖动大，体验不稳定
- 单点/单地域故障影响面大
- 成本与扩展性问题：流量集中、带宽峰值与回源成本高

现有 CDN/GA/PCDN 方案要么是闭源商用产品，要么难以在开源项目中同时兼顾：
- **个人开发者可快速上手**
- **企业可私有化部署**
- **可插拔扩展（入口/调度/鉴权/缓存/存储）**

本项目目标是提供一个开源框架，作为后续产品与新项目的“底座”。

---

## 2. 目标（Goals）

### 2.1 产品目标
1. **提供可运行的开源参考实现**：一键部署 demo（控制面 + 边缘节点 + 源站）。
2. **默认 302 调度可用**：支持下载/点播的 HTTP 对象分发（Range/断点续传）。
3. **调度可靠**：Top-K 候选 + 失败切换 + 节点探活，避免黑洞。
4. **可观测与可治理**：指标、事件、最小风控能力（防盗链/限流/隔离）。
5. **可扩展**：入口适配、调度策略、鉴权、缓存、存储均可插件化。

### 2.2 非目标（Non-Goals, v0.1）
以下不作为 v0.1 的强约束（可作为 Roadmap）：
- 真正 BGP Anycast 的全链路交付（可作为 PoP 入口增强）
- 复杂 P2P 打洞、家庭宽带大规模上传贡献网络
- 全面商业级计费、对账、结算系统
- 强一致性内容分发（以最终一致/缓存一致为主）

## 3. 目标用户与使用场景

### 3.1 用户画像（Personas）
1. **个人开发者**
   - 需求：快速搭建下载分发/加速 demo，用于个人项目或学习
   - 特点：资源少、部署简单优先
2. **C 端产品团队**
   - 需求：给 App/网站提供更稳定的多地域访问，加速下载/点播
   - 特点：更关注体验指标与自动化运维
3. **B 端/企业客户**
   - 需求：私有化部署、审计、安全策略、可与现有系统集成
   - 特点：更关注合规、可控、SLA、可观测

### 3.2 典型场景（Use Cases）
- UC1：对象下载加速（大文件、支持断点续传）
- UC2：点播文件分发（热点内容缓存命中）
- UC3：跨地域 API/静态资源加速（Roadmap）
- UC4：直播 HLS/DASH 分片分发（Roadmap）

## 4. 范围与模块（Scope）

### 4.1 v0.1 必含模块
- **Control Plane**
  - 节点注册/鉴权
  - 心跳/状态上报
  - 多探测点可达性探活（最小实现可为单探测点，接口预留）
  - 调度 API：输出 Top-K 候选 + token + ttl
  - 最小策略：黑名单/限流（可先做占位）
- **Edge Agent**
  - HTTP 服务（支持 Range）
  - 磁盘缓存 + 淘汰策略
  - 回源（源站/父节点）
  - 指标与上报
- **Ingress（默认 302）**
  - `GET /obj/{key}` -> 302 redirect（Location + 可选备选列表）

### 4.2 v0.2+（Roadmap）
- DNS/GSLB 适配器
- 网关反代适配器（NAT 节点/强控制）
- 内容索引（热内容精确 + 冷内容摘要）
- 策略与风控增强
- 直播分片策略、HTTP/3 等
- Anycast PoP 入口（可选增强）

## 5. 功能需求（Functional Requirements）

### FR-1 节点注册与身份
- 边缘节点可向控制面注册并获得唯一 `node_id`
- 节点与控制面通信需要鉴权（token 或 mTLS）
- 支持吊销/禁用节点

### FR-2 心跳与状态上报
- 节点周期性上报：资源使用、带宽、连接数、错误率、缓存命中率等
- 控制面将节点状态汇总为“可调度/不可调度/隔离”等状态机

### FR-3 可达性探活
- 控制面（或探测组件）对节点服务端点做可达性验证
- 探活结果参与调度决策（硬门槛或强权重）

### FR-4 调度（302 + Top-K）
- 控制面可根据请求（用户画像 + 资源）返回 Top-K 候选边缘节点
- 入口可返回 302 Location（首选）+ 备选列表（可选）
- 候选带 `ttl`，避免长时间粘住劣化节点

### FR-5 边缘缓存与回源
- 边缘节点支持缓存命中与回源
- 支持 Range：既支持客户端 Range，也支持对源站 Range 回源

### FR-6 Token/签名（最小防盗链）
- 支持短有效期 token，用于防盗链/防滥用（最小 HMAC 即可）
- token 至少绑定 `resource_key` 与 `exp`

### FR-7 可观测与事件
- 暴露指标（Prometheus 或等价）
- 输出事件（节点上下线、隔离/恢复、调度降级等）

---

## 6. 非功能需求（Non-Functional Requirements）

### NFR-1 易部署
- 提供 `docker compose` 一键 demo（v0.1 目标）

### NFR-2 可扩展性
- 控制面与边缘可水平扩展
- 插件机制与接口稳定

### NFR-3 可靠性与降级
- 无可达边缘节点时可降级到源站（或返回 503，策略可配）
- 调度与探活故障不应导致系统整体不可用（允许降级）

### NFR-4 安全
- 节点鉴权
- 调度 token
- 最小 ACL/限流能力（可先简化）

### NFR-5 可观测
- 指标：调度 QPS、命中率、失败率、RTT/探活成功率
- 日志可关联 request_id

## 8. 风险与对策（Risks）

- **NAT 节点不可入站**：v0.1 不强依赖；v0.3 引入网关/隧道能力
- **调度黑洞**（节点挂了仍被调度）：必须探活与自动剔除
- **滥用/刷量**：v0.1 最小 token + 限流；后续逐步风控
- **接口频繁变化**：先冻结最小核心 API，插件扩展而不是改动主接口

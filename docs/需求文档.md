# Edge Dispatch Framework

> 面向 B 端、C 端与个人开发者的开源边缘分发/加速框架：结合 **Anycast（可选入口增强）+ Edge 节点**，通过**中心调度（默认 302 重定向）**把请求分配到更合适的边缘节点，并提供可观测、策略与安全控制的基础能力。
>
> 状态：**设计与接口优先（WIP）**。本仓库当前以文档与协议为主，代码实现将按 Roadmap 推进。

## 你能用它做什么

- **下载/点播分发（优先支持）**：HTTP 对象分发、Range/断点续传、边缘缓存、回源。
- **直播分片分发（Roadmap）**：HLS/DASH 小分片高频请求的缓存与预取。
- **通用边缘代理（Roadmap）**：把边缘节点作为 L7/L4 代理与加速层（API 加速/动态回源等）。

## 核心概念（3 分钟理解）

- **控制面（Control Plane）**：节点注册、心跳、探活、调度决策、策略与审计。
- **数据面（Data Plane / Edge Agent）**：边缘节点的缓存/回源/服务能力，并向控制面上报状态。
- **入口（Ingress Adapter）**：把“调度结果”映射到实际接入方式。本项目默认实现思路为 **302 调度**，并支持扩展到 DNS/GSLB、统一网关反代等。
- **Top-K 候选**：调度一次返回多个候选边缘节点（3~5 个），客户端/SDK 可快速失败切换。

## 默认方案：302 调度（推荐用于 v0.1）

业务方访问统一入口：

```
https://dispatch.example.com/obj/<key>
```

控制面返回：
- `302 Location: https://edge-a.example.com/obj/<key>?token=...`
- 可选：`X-Alt-Location: https://edge-b... , https://edge-c...`

客户端直连边缘节点获取内容；失败时按候选列表切换，并支持 Range 断点续传。

## 部署形态（面向不同用户）

- **个人开发者**：单机 Docker / Compose 体验 demo；本地小规模边缘节点。
- **C 端产品团队**：在云上部署控制面 + 多地域边缘节点，按需扩展。
- **B 端/企业**：私有化部署（K8s/Helm），接入企业 IAM、日志与审计系统。

## 安全与合规（默认内置的方向）

- URL 签名 token（短有效期）用于防盗链与滥用控制
- 节点注册鉴权（建议 mTLS / 证书）
- 多探测点可达性验证（避免只信节点自报）

## Roadmap（建议优先级）

### v0.1（下载/点播最小可用）
- 控制面：注册/心跳/可达性探活/基础调度（健康 + 负载 + 就近）
- 边缘：HTTP Cache + 回源 + Range
- 入口：302 调度
- 可观测：Prometheus 指标（命中率、吞吐、失败率、在线）

### v0.2（可扩展与多入口）
- 内容索引（热内容精确表 + 冷内容摘要）
- 调度策略插件（命中优先/成本优先/风险隔离）
- DNS 适配器（GSLB）

### v0.3（复杂网络与 NAT 节点）
- 统一网关反代模式（适配 NAT/入站不可达节点）
- 反向隧道/中继（可选模块）

### v0.4（直播与更高性能）
- HLS/DASH 分片策略（预取、滑窗缓存）
- HTTP/3/QUIC（可选）

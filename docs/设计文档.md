# 设计文档（详细设计）：Edge Dispatch Framework

> 版本：v0.1（草案）  
> 本文是对 `architecture.md` 的落地细化：模块边界、数据模型、状态机、插件点、部署与安全建议。  
> 默认入口：**302 调度**；Anycast 作为 **可选 PoP 入口增强**（不作为 v0.1 前置条件）。

---

## 1. 术语与约束

- **Edge**：边缘节点（数据面），提供缓存/回源/服务能力。
- **Control Plane**：控制面，负责节点管理、探活、调度与策略。
- **Ingress Adapter**：入口适配（302/DNS/网关），把调度结果映射为实际接入行为。
- **Top-K**：调度返回多个候选节点，客户端/SDK 做快速失败切换。
- **NAT Node**：入站不可达节点；v0.1 不作为公网服务节点。

核心约束（v0.1）：
- 以 **HTTP 对象分发** 为主；必须支持 **Range/断点续传**。
- 调度必须依赖 **探活/可达性**，避免黑洞。
- 所有关键行为可观测（指标 + request_id）。

---

## 2. 模块拆分（建议实现单元）

### 2.1 Control Plane 服务拆分

> v0.1 可合并为一个进程；后续可按模块拆分为独立服务。

- **cp-api**：对外 REST API（Dispatch/Node/Admin）
- **cp-registry**：节点注册、凭证签发、吊销
- **cp-heartbeat**：心跳接收、节点状态聚合
- **cp-prober**：可达性探测（主动探活）
- **cp-scheduler**：调度引擎（Top-K）
- **cp-policy**：ACL、限流、成本、租户隔离（最小实现可只做黑名单）
- **cp-event**：事件流（节点上下线/隔离/恢复）

### 2.2 Edge Agent 组件
- **edge-server**：HTTP 服务层（Range、限速、TLS 可选）
- **edge-cache**：缓存管理（磁盘为主）
- **edge-fetcher**：回源（Origin/Parent），支持 Range 回源
- **edge-reporter**：心跳/指标/摘要上报
- **edge-health**：本地健康检查（端口、磁盘、回源可用性）

### 2.3 Ingress Adapter（入口适配）
- **302 Adapter（默认）**：返回 `Location` + 备选列表（可选）
- **DNS Adapter（Roadmap）**
- **Gateway Adapter（Roadmap）**
- **Anycast PoP（可选增强）**：用于“把用户带到最近入口 PoP”，PoP 内再调度到 Edge/Parent

## 3. 数据模型（最小集合）

> 这里是“框架底座”的关键：后续新项目可以复用这些表/结构。

### 3.1 Node（节点）
- `node_id`：全局唯一
- `tenant_id`：多租户隔离
- `name`：展示名
- `endpoints[]`：对外服务端点（scheme/host/port）
- `region` / `isp` / `asn`：就近与策略维度
- `capabilities`：
  - `inbound_reachable`（是否可作为公网服务节点）
  - `cache_disk_gb`
  - `max_uplink_mbps`
  - `supports_https` 等
- `status`：见 3.3 状态机
- `last_seen_at`
- `scores`：
  - `reachable_score`（来自 prober）
  - `health_score`（来自心跳与失败率）
  - `risk_score`（风控/作弊风险）

### 3.2 Resource（资源）
v0.1 可不建“资源主表”，只把 `resource_key` 当作字符串；v0.2 引入：
- `key`
- `type`（object / chunk）
- `size`（可选）
- `etag/hash`（可选）

### 3.3 Node 状态机（建议）

状态：
- `REGISTERED`：已注册未验证
- `ACTIVE`：在线且可调度
- `DEGRADED`：在线但质量下降（降权）
- `QUARANTINED`：隔离（不调度）
- `OFFLINE`：离线

迁移触发（示例）：
- 心跳超时 -> OFFLINE
- 探活失败率超过阈值 -> DEGRADED / QUARANTINED
- 恢复探活与低错误率持续一段时间 -> ACTIVE

## 4. 调度设计（Scheduler）

### 4.1 调度输入
- client：IP/region/isp/asn（可由服务端推断）
- resource：key、类型（v0.1 object）、scheme
- hints：是否需要 Range、是否允许降级到源站、期望协议等

### 4.2 过滤与排序（推荐两阶段）
1. **Hard Filter（硬过滤）**
   - status 必须为 ACTIVE/DEGRADED（可配置）
   - reachable_score >= threshold
   - inbound_reachable == true（v0.1 默认要求）
   - 策略允许（黑名单/ACL）
2. **Scoring（打分排序）**
   - proximity（region/isp/asn 匹配）
   - spare_capacity（带宽余量、连接余量）
   - health（错误率、近期失败）
   - content_hit（v0.2 引入）
   - cost（v0.2 引入）
   - risk（隔离/降权）

输出：
- Top-K candidates（3~5）
- ttl_ms（建议 10~60s）
- token（短有效期）
- client_hints（重试建议）

### 4.3 降级策略（v0.1）
当无可用边缘节点时：
1) 若允许降级：返回源站（或父节点）URL 作为候选  
2) 否则：返回 `EDGE_NOT_FOUND`（503）

## 5. 可达性探测（Prober）

### 5.1 探测方式（v0.1）
- TCP connect 探测 `host:port`
- HTTPS HEAD/GET 探测健康路径（如 `/healthz`）

### 5.2 探测输出
为每个 endpoint 维护：
- `success_rate_1m/5m`
- `rtt_p50/p95`
- `last_ok_at`

并聚合为节点级 `reachable_score`。

---

## 6. Token/签名（Auth Plugin）

### 6.1 v0.1 建议：HMAC Query Token
签名内容建议包含：
- `key`
- `exp`
- 可选：`ip_prefix`

示例（概念）：
```
token = base64url(hmac_sha256(secret, key + "." + exp + "." + ip_prefix))
```

### 6.2 插件化接口（建议）
- `Sign(resource, client, ttl) -> token`
- `Verify(request) -> allow/deny + reason`

---

## 7. Edge 缓存与回源（Cache/Fetcher）

### 7.1 缓存键
- `cache_key = resource_key + (variant)`（variant 可包含编码/协议等）

### 7.2 Range 支持
必须同时支持：
- 客户端 Range（206）
- 回源 Range（从源站按 Range 拉取缺失片段）

> v0.1 可先“回源整文件缓存 + Range 切片输出”；后续优化为“分片缓存/稀疏文件”。

### 7.3 淘汰
v0.1：LRU（按最后访问时间）+ 磁盘水位线触发。

---

## 8. 可观测性（Observability）

### 8.1 指标（建议最小集合）
控制面：
- dispatch_qps
- dispatch_no_candidate_total
- candidate_count_histogram

边缘：
- edge_requests_total（按状态码分维度）
- edge_cache_hit_ratio
- edge_origin_fetch_latency
- edge_egress_bytes

探活：
- probe_success_rate
- probe_rtt_ms

### 8.2 日志与关联
- 每次调度生成 `request_id`，写入响应 header（例如 `X-Request-Id`）
- 边缘访问日志携带该 id（若客户端透传）或生成新的 trace id
